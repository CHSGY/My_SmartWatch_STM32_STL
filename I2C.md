# I2C协议在智能手表项目中的应用

## 一、项目概述

本项目基于 **STM32F103C8** (Cortex-M3, 72MHz)，使用 **软件模拟I2C (Bit-Banging)** 方式驱动两个I2C设备：

| 设备 | 型号 | 7位地址 | 8位写地址 | 8位读地址 | 用途 |
|------|------|---------|-----------|-----------|------|
| OLED显示屏 | SSD1306 | 0x3C | 0x78 | 0x79 | 显示界面 |
| 六轴传感器 | MPU6050 | 0x68 | 0xD0 | 0xD1 | 姿态检测 |

---

## 二、硬件设计

### 2.1 引脚分配

```
┌─────────────────────────────────────────────────────────┐
│                    STM32F103C8                          │
├─────────────────────┬───────────────────────────────────┤
│   OLED I2C总线      │        MPU6050 I2C总线            │
│   PB8  → SCL        │        PB10 → SCL                │
│   PB9  → SDA        │        PB11 → SDA                │
└─────────────────────┴───────────────────────────────────┘
```

### 2.2 为什么选择软件I2C？

| 对比项 | 硬件I2C | 软件I2C |
|--------|---------|---------|
| 引脚限制 | 固定引脚 | 任意GPIO |
| 时序控制 | 硬件自动 | 软件可控 |
| 调试难度 | 较难 | 简单直观 |
| 多设备支持 | 需多个外设 | 灵活扩展 |

**本项目选择软件I2C的原因**：STM32F103硬件I2C存在已知BUG（参考勘误表），软件实现更可靠。

### 2.3 GPIO配置要求

```c
// 必须配置为开漏输出模式
GPIO_InitStructure.GPIO_Mode = GPIO_Mode_Out_OD;  // 关键！
```

**为什么是开漏输出？**
- 支持 **线与** 逻辑：多个设备可共享总线
- 支持 **电平转换**：通过外部上拉电阻可连接不同电压设备
- 避免 **总线冲突**：输出0时拉低，输出1时浮空（由上拉电阻拉高）

## 四、I2C协议时序实现

### 4.1 起始信号 (Start)

```
SCL:  ‾‾‾‾‾‾╲_____
SDA:  ‾‾‾╲________
         ↑
    SCL高电平时，SDA下降沿
```

```c
void MyI2C_Start(void)
{
    MyI2C_W_SDA(1);   // 释放SDA
    MyI2C_W_SCL(1);   // 释放SCL
    MyI2C_W_SDA(0);   // SCL高电平时拉低SDA → 起始
    MyI2C_W_SCL(0);   // 拉低SCL，占用总线
}
```

### 4.2 停止信号 (Stop)

```
SCL:  _____╱‾‾‾‾‾
SDA:  _____╱‾‾‾‾‾
              ↑
    SCL高电平时，SDA上升沿
```

```c
void MyI2C_Stop(void)
{
    MyI2C_W_SDA(0);   // 拉低SDA
    MyI2C_W_SCL(1);   // 释放SCL
    MyI2C_W_SDA(1);   // SCL高电平时释放SDA → 停止
}
```

### 4.3 数据传输

**发送一个字节** (MSB先发)：
```c
void MyI2C_SendByte(uint8_t Byte)
{
    uint8_t i;
    for (i = 0; i < 8; i++)
    {
        MyI2C_W_SDA(!!(Byte & (0x80 >> i)));  // 从高位到低位
        MyI2C_W_SCL(1);   // SCL高电平，从机读取SDA
        MyI2C_W_SCL(0);   // SCL低电平，准备下一位
    }
}
```

**接收一个字节**：
```c
uint8_t MyI2C_ReceiveByte(void)
{
    uint8_t i, Byte = 0x00;
    MyI2C_W_SDA(1);  // 主机释放SDA
    for (i = 0; i < 8; i++)
    {
        MyI2C_W_SCL(1);
        if (MyI2C_R_SDA()) { Byte |= (0x80 >> i); }
        MyI2C_W_SCL(0);
    }
    return Byte;
}
```

### 4.4 应答机制 (ACK/NACK)

```
发送方:  ──────╲_____╱‾‾‾‾‾
接收方:  ════════[ACK]══════
                 ↑
           接收方拉低SDA = 应答
```

```c
// 发送应答：0=应答, 1=非应答
void MyI2C_SendAck(uint8_t AckBit)
{
    MyI2C_W_SDA(AckBit);
    MyI2C_W_SCL(1);
    MyI2C_W_SCL(0);
}

// 接收应答：返回0表示应答成功
uint8_t MyI2C_ReceiveAck(void)
{
    uint8_t AckBit;
    MyI2C_W_SDA(1);      // 释放SDA
    MyI2C_W_SCL(1);
    AckBit = MyI2C_R_SDA();
    MyI2C_W_SCL(0);
    return AckBit;
}
```

## 五、设备驱动实现

### 5.1 MPU6050读写寄存器

**写寄存器流程**：
```
Start → 从机地址(W) → ACK → 寄存器地址 → ACK → 数据 → ACK → Stop
```

```c
void MPU6050_WriteReg(uint8_t RegAddress, uint8_t Data)
{
    MyI2C_Start();
    MyI2C_SendByte(MPU6050_ADDRESS);      // 0xD0
    MyI2C_ReceiveAck();
    MyI2C_SendByte(RegAddress);
    MyI2C_ReceiveAck();
    MyI2C_SendByte(Data);
    MyI2C_ReceiveAck();
    MyI2C_Stop();
}
```

**读寄存器流程** (使用重复起始信号)：
```
Start → 从机地址(W) → ACK → 寄存器地址 → ACK → 
Start(重复) → 从机地址(R) → ACK → 数据 → NACK → Stop
```

```c
uint8_t MPU6050_ReadReg(uint8_t RegAddress)
{
    uint8_t Data;
    MyI2C_Start();
    MyI2C_SendByte(MPU6050_ADDRESS);      // 先写寄存器地址
    MyI2C_ReceiveAck();
    MyI2C_SendByte(RegAddress);
    MyI2C_ReceiveAck();
    
    MyI2C_Start();                         // 重复起始！
    MyI2C_SendByte(MPU6050_ADDRESS | 0x01); // 读模式
    MyI2C_ReceiveAck();
    Data = MyI2C_ReceiveByte();
    MyI2C_SendAck(1);                      // 发送NACK，停止接收
    MyI2C_Stop();
    return Data;
}
```

### 5.2 OLED写命令/数据

OLED使用 **Co位 + DC位** 控制字节区分命令和数据：

```c
// 从机地址 0x78
// 控制字节: 0x00=写命令, 0x40=写数据

void OLED_WriteCommand(uint8_t Command)
{
    OLED_I2C_Start();
    OLED_I2C_SendByte(0x78);    // 从机地址
    OLED_I2C_SendByte(0x00);    // 控制字节：命令
    OLED_I2C_SendByte(Command);
    OLED_I2C_Stop();
}

void OLED_WriteData(uint8_t *Data, uint8_t Count)
{
    OLED_I2C_Start();
    OLED_I2C_SendByte(0x78);
    OLED_I2C_SendByte(0x40);    // 控制字节：数据
    for (uint8_t i = 0; i < Count; i++)
    {
        OLED_I2C_SendByte(Data[i]);
    }
    OLED_I2C_Stop();
}
```

## 六、关键设计要点

### 6.1 时序控制

```c
void MyI2C_W_SCL(uint8_t BitValue)
{
    GPIO_WriteBit(GPIOB, GPIO_Pin_10, (BitAction)BitValue);
    Delay_us(10);  // 每次操作后延时10us
}
```

- **MPU6050总线**：每次GPIO操作后延时10us → SCL频率约50kHz
- **OLED总线**：无显式延时，依赖MCU执行速度

### 6.2 两组独立I2C总线

| 总线 | SCL | SDA | 延时 | 设备 |
|------|-----|-----|------|------|
| I2C1 | PB10 | PB11 | 10us | MPU6050 |
| I2C2 | PB8 | PB9 | 无 | OLED |

**独立总线的好处**：
- 避免设备地址冲突
- 可独立控制通信速率
- 一个设备故障不影响另一个

## 七、面试常见问题

### Q1: 为什么用软件I2C不用硬件I2C？
**A**: 
1. STM32F10x硬件I2C存在已知BUG（参考勘误表）
2. 软件I2C引脚更灵活，不局限于固定引脚
3. 本项目I2C设备少，速度要求不高，软件实现足够

### Q2: 为什么GPIO要配置为开漏输出？

#### 核心原因：支持"线与"逻辑，避免总线冲突

---

#### 2.1 什么是开漏输出？

```
开漏输出内部结构：
                    VCC
                     │
                     R (上拉电阻，外部接)
                     │
    ┌────────────────┼─────────────── SDA/SCL总线
    │                │
    │    ┌───────────┴───────────┐
    │    │      开漏管           │
    │    │    ┌─────┐            │
    │    │    │ MOS │            │
    │    │    └──┬──┘            │
    │    └───────┼───────────────┘
    │            │
    └────────────┴─────────────────── GND
```

**开漏输出的特点**：
| MOS管状态 | 输出电平 | 总线状态 |
|-----------|----------|----------|
| 导通 (输出0) | 低电平 | 被拉低到GND |
| 截止 (输出1) | 高阻态 | 由上拉电阻拉高到VCC |

**对比推挽输出**：
| 输出类型 | 输出0 | 输出1 |
|----------|-------|-------|
| 推挽输出 | 主动拉低 | 主动推高 |
| 开漏输出 | 主动拉低 | **浮空**（靠上拉电阻） |

---

#### 2.2 什么是"线与"逻辑？

**线与 (Wired-AND)**：多个设备连接到同一总线时，只要有一个设备输出低电平，总线就是低电平。

```
示例：两个设备共享SDA线

         VCC (3.3V)
          │
          R (上拉电阻)
          │
    ┌─────┴─────┬─────────────┐
    │           │             │
    │           │             │
┌───┴───┐   ┌───┴───┐        │
│ 设备A │   │ 设备B │        │
│(STM32)│   │(MPU6050)│       │
│ 开漏  │   │  开漏  │        │
└───┬───┘   └───┬───┘        │
    │           │             │
    └─────┬─────┘             │
          │                   │
         GND                 GND

线与真值表：
┌────────┬────────┬────────────┐
│ 设备A  │ 设备B  │  总线电平   │
├────────┼────────┼────────────┤
│ 输出0  │ 输出0  │    0 (低)   │
│ 输出0  │ 输出1  │    0 (低)   │  ← 只要有一个低，总线就低
│ 输出1  │ 输出0  │    0 (低)   │  ← 只要有一个低，总线就低
│ 输出1  │ 输出1  │    1 (高)   │  ← 都浮空，上拉电阻拉高
└────────┴────────┴────────────┘

逻辑表达式：总线 = A AND B（线与）
```

---

#### 2.3 什么是总线冲突？举例说明

**总线冲突 (Bus Contention)**：多个设备同时驱动总线，一个输出高，一个输出低，导致短路。

##### 反例：如果用推挽输出会怎样？

```
场景：STM32想发高电平，MPU6050想发低电平

         VCC (3.3V)
          │
    ┌─────┴─────┐
    │           │
┌───┴───┐   ┌───┴───┐
│ STM32 │   │MPU6050│
│推挽输出│   │推挽输出│
└───┬───┘   └───┬───┘
    │           │
    │  输出1    │  输出0
    │  (推高)   │  (拉低)
    │           │
    ├─── 3.3V   ├─── 0V
    │           │
    └─────┬─────┘
          │
    ┌─────┴─────┐
    │  短路！！  │  ← 电流从VCC直接流到GND
    └───────────┘

结果：
1. 大电流流过两个输出级 → 可能烧毁IO口
2. 总线电平不确定 → 通信失败
```

##### 正例：开漏输出如何避免冲突？

```
场景：STM32想发高电平(输出1)，MPU6050想发低电平(输出0)

         VCC (3.3V)
          │
          R (上拉电阻)
          │
    ┌─────┴─────┐
    │           │
┌───┴───┐   ┌───┴───┐
│ STM32 │   │MPU6050│
│开漏输出│   │开漏输出│
└───┬───┘   └───┬───┘
    │           │
    │  输出1    │  输出0
    │  (浮空)   │  (拉低)
    │           │
    │   高阻    ├─── 0V
    │           │
    └─────┬─────┘
          │
          0V (被MPU6050拉低)

结果：
1. 无短路：STM32是高阻态，不主动驱动总线
2. 总线电平确定：被MPU6050拉低到0V
3. 通信正常：STM32可以读取到低电平
```

#### 2.4 实际I2C通信中的线与应用

##### 应用1：从机应答 (ACK)

```
主机发送完一个字节后，需要读取从机的应答：

时序：
SCL:  _____╱‾╲__╱‾╲__╱‾╲__
SDA:  ═══════════════╲______  ← 从机拉低SDA表示ACK
                      ↑
              从机输出0，主机输出1(浮空)

过程：
1. 主机释放SDA（输出1 = 高阻态）
2. 从机拉低SDA（输出0）
3. 线与结果：总线 = 1 AND 0 = 0（低电平）
4. 主机读取到ACK
```

##### 应用2：多主机仲裁

```
场景：两个主机同时想发送数据

主机A想发: 1 0 1 1 ...
主机B想发: 1 0 0 1 ...
                ↑
            第3位不同

仲裁过程：
┌──────┬──────┬────────┬───────────────────┐
│ 位   │ 主机A│ 主机B  │ 总线 (线与)        │
├──────┼──────┼────────┼───────────────────┤
│ 第1位│  1   │   1    │  1 (都浮空)        │
│ 第2位│  0   │   0    │  0 (都拉低)        │
│ 第3位│  1   │   0    │  0 (B拉低)         │ ← 仲裁点
└──────┴──────┴────────┴───────────────────┘

第3位分析：
- 主机A输出1（浮空），期望读到1
- 主机B输出0（拉低），实际总线是0
- 主机A读取总线发现是0，与自己发的1不同
- 主机A发现自己输了仲裁，主动退出，让主机B继续发送

这就是线与的妙处：低电平"优先级更高"
```

#### 2.5 总结：开漏输出的三大优势

| 优势 | 说明 | 本项目应用 |
|------|------|------------|
| **线与逻辑** | 多设备共享总线，低电平优先 | MPU6050和STM32共享I2C总线 |
| **避免冲突** | 高阻态+上拉，不会短路 | 主机和从机可安全切换SDA方向 |
| **电平转换** | 外部上拉可接不同电压 | 如需接5V设备，上拉到5V即可 |

```
电平转换示例：
    5V                      3.3V
     │                        │
     R1                       R2
     │                        │
     ├───── SDA总线 ──────────┤
     │                        │
  ┌──┴──┐                 ┌───┴──┐
  │ 5V  │                 │ 3.3V │
  │设备 │                 │ MCU  │
  └─────┘                 └──────┘

两边都是开漏输出，总线电平由上拉电阻决定
- 5V设备看总线：0V=低，5V=高
- 3.3V MCU看总线：0V=低，3.3V=高
- 电平转换自然完成！
```

### Q3: 读操作为什么要发两次Start？
**A**: 
第一次Start+写：告诉从机要读哪个寄存器  
第二次Start+读：切换为读模式，获取数据  
这叫 **重复起始信号 (Repeated Start)**，避免释放总线后被其他主机抢占

### Q4: 读取最后一个字节为什么要发NACK？
**A**: 
ACK表示"收到了，继续发"  
NACK表示"收到了，停止发送"  
最后一个字节发NACK告诉从机传输结束

### Q5: I2C的时序要求是什么？
**A**:
- SCL高电平期间，SDA必须保持稳定（数据有效）
- SDA只能在SCL低电平时变化
- 起始信号：SCL高时SDA下降沿
- 停止信号：SCL高时SDA上升沿
